## Введение

Архитектурный стиль REST (Representational State Transfer) стал стандартом де-факто для проектирования веб-API благодаря своей простоте, масштабируемости и совместимости с существующими веб-стандартами. REST основан на принципах, предложенных Роем Филдингом в 2000 году, и широко применяется для построения клиент-серверных приложений, особенно в контексте микросервисной архитектуры и мобильных приложений.

Важным аспектом разработки RESTful API является его документирование. Инструменты OpenAPI (ранее Swagger) позволяют формально описать контракт API, автоматизировать генерацию клиентских и серверных кодов, а также интерактивно тестировать эндпоинты.

## Принципы REST

REST опирается на шесть ключевых принципов:

1.  **Единообразие интерфейса** – стандартизация взаимодействия через HTTP-методы (GET, POST, PUT, DELETE) и использование ресурсов в виде URI.
2.  **Отсутствие состояния** – каждый запрос содержит всю информацию, необходимую для его обработки.
3.  **Кешируемость** – ответы должны явно указывать, можно ли их кешировать
4.  **Клиент-серверная архитектура** – разделение ответственности между клиентом и сервером;
5.  **Слоистая система** – возможность введения промежуточных слоёв (прокси, балансировщики) без изменения клиентского кода.
6.  **Код по требованию (опционально)** – возможность передачи исполняемого кода клиенту (например, JavaScript).

## Лучшие практики проектирования REST API

Таблица 1. Рекомендации по проектированию RESTful API

| Аспект                  | Рекомендация                                                                 |
|-------------------------|------------------------------------------------------------------------------|
| Именование ресурсов     | Существительные во мн. числе: `/users`, `/orders`                            |
| HTTP-методы             | GET → чтение, POST → создание, PUT/PATCH → обновление, DELETE → удаление    |
| Статус-коды             | 200, 201, 204, 400, 401, 403, 404, 429, 500 и т.д.                           |
| Версионирование         | В URI (`/api/v1/users`) или в заголовках                                     |
| Пагинация и фильтрация  | `limit`, `offset`, `page`, `sort`, `filter`                                  |
| Безопасность            | HTTPS + OAuth2/JWT + валидация и санитизация входных данных                  |

## Документирование API c использованием OpenAPI/Swagger

OpenAPI – это стандарт для описания RESTful API, который позволяет формально определить.

- эндпоинты и методы;
- параметры запросов и ответов/
- модели данных;
- аутентификацию.

## Преимущества использования OpenAPI

- Автоматическая генерация интерактивной документации (Swagger UI, ReDoc);
- Генерация клиентского и серверного кода на десятках языков;
- Возможность валидации запросов/ответов на этапе разработки и runtime;
- Упрощение взаимодействия между фронтенд- и бэкенд-командами благодаря единому контракту.

**Пример минимального OpenAPI 3.0 документа (YAML):**

```yaml
openapi: 3.0.0
info:
  title: User API
  version: 1.0.0
paths:
  /users:
    get:
      summary: Получить список всех пользователей
      responses:
        '200':
          description: OK
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/User'
components:
  schemas:
    User:
      type: object
      properties:
        id:
          type: integer
          example: 42
        name:
          type: string
          example: Иван Иванов
        email:
          type: string
          format: email
```

## Заключение

REST остаётся одним из наиболее популярных и устойчивых архитектурных стилей благодаря своей простоте, совместимости с протоколом HTTP и удобству интеграции. Соблюдение шести фундаментальных принципов и современных лучших практик позволяет создавать масштабируемые, понятные и надёжные API.

Использование спецификации OpenAPI (ранее Swagger) для документирования значительно ускоряет процесс разработки, улучшает коммуникацию в команде и обеспечивает согласованность между клиентской и серверной частями приложения. В современных условиях, когда API становятся ключевым элементом интеграции систем, качественное и всегда актуальное документирование является неотъемлемой частью жизненного цикла разработки.
